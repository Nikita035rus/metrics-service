# Metrics Service (АО «МТП»)

REST-сервис для приема, хранения и агрегации метрик. Реализован с использованием Spring Boot 3, JWT авторизации и распределенного Rate Limiting на базе Redis.

## Технологический стек
*   **Java 21** (LTS)
*   **Spring Boot 3.4.3**
*   **Spring Data JPA** (хранение метрик)
*   **Spring Security + JJWT** (авторизация)
*   **Spring AOP** (реализация Rate Limiting через аспекты)
*   **Redis** (хранилище для счетчиков Rate Limit)
*   **H2 Database** (реляционная БД в памяти для тестового задания)
*   **Lombok** (сокращение шаблонного кода)
*   **JUnit 5 / Mockito** (интеграционное тестирование)

---

## Архитектура решения

1.  **Слой контроллеров**: Обрабатывает входящие HTTP-запросы, выполняет валидацию данных (@Valid) и обеспечивает безопасность через Spring Security.
2.  **Слой аспектов (AOP)**: Кастомная аннотация @RateLimit позволяет декларативно задавать лимиты на методы. Аспект перехватывает вызов, проверяет счетчики в Redis и выбрасывает RateLimitExceededException при превышении.
3.  **Слой сервисов**: Содержит бизнес-логику сохранения метрик и подготовки агрегированных данных.
4.  **Слой данных**: Использует Spring Data JPA. Агрегация (COUNT, AVG, MIN, MAX) вынесена на уровень БД через JPQL-запрос, что предотвращает загрузку лишних данных в память и повышает производительность.

## Работа с датами и часовыми поясами
Для работы с временными метками используется тип OffsetDateTime.
*   **Прием данных**: API принимает даты в формате ISO 8601 (например, 2024-01-15T10:30:00+03:00). Это гарантирует корректное сохранение момента времени с учетом смещения часового пояса клиента.
*   **Хранение**: В БД используется тип TIMESTAMP WITH TIME ZONE, что исключает искажение данных при их анализе из разных часовых поясов.
*   **Фильтрация**: При запросе статистики параметры from и to также учитывают смещение, позволяя точно ограничивать выборку.

---

## Реализация Rate Limiting
Лимитирование реализовано через Spring AOP + Redis:
*   **Атомарность**: Используется атомарная команда Redis INCR, что гарантирует корректный подсчет даже при конкурентных запросах.
*   **TTL**: Для каждого счетчика устанавливается время жизни (1 секунда) через команду EXPIRE.
*   **Типы лимитов**:
    *   local: Лимит на конкретный метод для каждого clientID.
    *   global: Общий лимит на метод для всех клиентов.
*   **Масштабируемость**: Использование внешнего хранилища (Redis) позволяет Rate Limiter-у работать синхронно на любом количестве запущенных экземпляров сервиса.

---

## Выбор БД и Агрегация
*   **Выбор**: Для тестового задания решил не разворачивать тяжелую Postgres и ограничился H2 в памяти, чтобы проект запускался одной кнопкой
*   **Агрегация**: Расчет статистики выполняется на стороне СУБД. Приложение получает готовый объект MetricsStatsResponse, содержащий только финальные цифры, что критично при работе с большими объемами данных.

## Инструкция по запуску

### Сборка проекта
Для сборки JAR-файла выполните:
mvn clean package

*В процессе сборки будут запущены тесты, проверяющие работу JWT, Rate Limiting (используются моки Redis для автономности).*

### Запуск через Docker Compose
Запуск приложения вместе с Redis (БД H2 запустится внутри приложения):
docker-compose up --build -d
* **Примечание:** В Windows можно столкнуться с проблемой, не видит Dockerfile. 
* **Решение:** Запуск командой: docker build -t metrics-service-app -f Dockerfile .
*   **Приложение**: http://localhost:8080
*   **H2 Console**: http://localhost:8080/h2-console (JDBC URL: jdbc:h2:mem:metricsdb, User: admin, Pass: пустой)

## 📝 Примеры использования

### 1. Получение JWT
curl -X POST http://localhost:8080/api/v1/auth \
-H "Content-Type: application/json" \
-d '{"clientId": "client-1"}'

### 2. Отправка метрики
curl -X POST http://localhost:8080/api/v1/metrics \
-H "Authorization: Bearer <ВАШ_ТОКЕН>" \
-H "Content-Type: application/json" \
-d '{
"timestamp": "2024-01-15T12:00:00+03:00",
"value": 45.5,
"payload": {"status": "ok", "app": "metrics-v1"}
}'

### 3. Получение статистики
curl -G http://localhost:8080/api/v1/metrics \
-H "Authorization: Bearer <ВАШ_ТОКЕН>" \
--data-urlencode "from=2024-01-15T00:00:00Z" \
--data-urlencode "to=2024-01-16T00:00:00Z"
